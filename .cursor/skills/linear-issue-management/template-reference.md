# Issue Template Reference

## The Template

Every issue created by the agent must match this structure. The template is rooted in INCOSE systems engineering discipline: needs trace to requirements, requirements trace to specifications. If a need doesn't justify the issue, delete the issue.

### Title

A concise description of what this issue accomplishes. Not a verb phrase ("Implement X") but a noun phrase or outcome ("Automated corpus delivery from object storage").

### Why? — the business problem that this issue is solving

Three sub-sections, always in this order:

**Business problem (1 paragraph):** What customer, revenue, risk, or strategic problem exists today? This should explain why this work deserves time over everything else we could be doing.

**Affected stakeholders (1 comma-separated list):** Which stakeholders — internal, external, or user personas — are affected by this problem? Per INCOSE systems engineering discipline, stakeholders are groups of natural persons. Never abstract concepts.

**Proposed solution (1 sentence):** What do you think needs to be done to solve the business problem?

### What? — how will we tell that this issue is solved

> This defines *success*, not tasks or implementation.

Three sub-sections of checkboxes:

**What must be true when this is done?** Observable conditions that define completion.

**What observable outcome, capability, or decision does this enable?** What becomes possible after this is done?

**How will we know it worked?** Metrics, behaviour, test protocols.

Followed by the acknowledgment:

- [x] I acknowledge that anyone can read just the text of this issue and understand why we're doing the work, what success looks like, and how it serves the business.

## Good Example

```yaml
who: alice@INF
title: "Enable static content delivery from object storage — serve directly from public bucket or establish a working manual path"
priority: 1
related:
  - INF-40
  - INF-66
why: |
  **Business problem:**

  There is no way to get generated content from object storage onto the servers that serve it. Content is generated by the pipeline and lands in a GCS bucket, but the last mile — actually serving it to clients — has no solution. Engineers have used scp manually. FUSE mounts are unreliable for many small files under load. The content team needs a path that works before the customer goes live.

  The fallback is manual scp with documented steps so the content engineer can do it without pinging infrastructure. Either path is acceptable as long as content is servable before the deadline.

  **Affected stakeholders:** Content engineers (cannot deliver generated content to serving nodes), customers in the deployment pipeline (content exists but no way to serve it), infrastructure engineers (scp does not scale past a handful of customers)

  **Proposed solution:** Either configure the object storage bucket as a public origin behind a load balancer or document a manual scp procedure that the content engineer can execute independently — whichever can ship by the deadline.
what: |
  > This defines *success*, not tasks or implementation.

  **What must be true when this is done?**
  - [ ] HTML files stored in the object storage bucket are servable to clients with correct MIME types
  - [ ] The content engineer can push new content to the bucket and have it appear on the serving path without pinging infrastructure — either automatically or via a documented self-service procedure
  - [ ] The solution works for both deployment modes the system supports

  **What observable outcome, capability, or decision does this enable?**
  - [ ] The content engineer generates content, pushes to storage, and it's served — the deployment is unblocked
  - [ ] We have a content delivery path that doesn't require infrastructure intervention for each customer

  **How will we know it worked?**
  - [ ] Upload a test HTML file to the bucket; curl it from the serving path; confirm HTML is returned with correct Content-Type
  - [ ] The content engineer performs the full flow (generate, push, verify served) without help from infrastructure

  ---
  - [x] I acknowledge that anyone can read just the text of this issue and understand why we're doing the work, what success looks like, and how it serves the business.
```

### What makes this good

- Business problem is one paragraph, focused on a single pain
- Stakeholders are people: "content engineers", "customers in deployment pipeline", "infrastructure engineers"
- Proposed solution is one sentence with concrete alternatives
- What section has checkboxes with observable outcomes, not implementation tasks
- "How will we know it worked" has concrete test steps anyone can run
- No mid-sentence line breaks
- Each bullet on its own line

## Bad Example

```yaml
who: sara@FE
title: "Admin dashboard for operators"
priority: 2
builds_on:
  - 2026-03-16-C1-01
why: |
  From SRS R16: an operator MUST be able to create a new tenant for a
  customer being onboarded. An operator SHOULD have clear indication about
  the state of said deployment.
what: |
  - [ ] Design tenant list view: show all tenants with status indicators
  - [ ] Design tenant editor: form-based editing with validation
  - [ ] Implement create flow: guided wizard, generates config, writes to bucket
  - [ ] Implement edit flow: load existing config, modify, validate, save
  - [ ] Implement delete flow: elevated friction
  - [ ] Deployment state indication
  - [ ] Testing instructions panel
  - [ ] Storage integration: read/write config files to bucket
  - [ ] Authentication: restrict access to authorized operators
```

### What makes this bad

- Why section quotes SRS requirements instead of explaining the business problem in plain language
- No "Business problem" / "Affected stakeholders" / "Proposed solution" structure
- Stakeholders not mentioned at all
- What section is a task list ("Design X", "Implement Y"), not success criteria
- No "What must be true" / "What does this enable" / "How will we know" structure
- No acknowledgment checkbox
- `builds_on` references a candidate file name instead of a Linear identifier — this causes a hard error in `easy-send`
- Mid-sentence line breaks throughout

## Stakeholder Anti-Patterns

Never write stakeholders as abstract concepts. Always name groups of people.

| Bad (abstract concept) | Good (group of persons) |
|------------------------|------------------------|
| Customer onboarding operations | Sales team (cannot self-service onboarding) |
| Product delivery velocity | Engineers building the dashboard (blocked) |
| Deployment reliability | Deployment engineers (guessing which variant to deploy) |
| Incident forensics | On-call engineers during incidents (zero replay capability) |
| Financial operations | Finance team (undetected cost anomalies) |
| Operational scalability | Content and infrastructure engineers (manual process does not scale) |
| Brand safety | Customer marketing and PR teams (corrupted content misrepresents their brand) |
| Classification rule consistency | Compiler engineers (cannot target a stable format) |
